### Fix 4: 红点徽标双约束条件

**PRD 章节**: 3.1、3.3 候选生成触发条件

**问题描述**:
红点徽标只检查第一约束（≥3 已确认事实），未检查第二约束（≥1 待确认共识）。

**修复内容**:
```javascript
function updateCandidateBadge() {
  // 第一约束：≥3 条已确认事实
  const confirmedFacts = state.records.filter(r => r.type === 'fact' && r.status === 'confirmed');
  const firstConstraintMet = confirmedFacts.length >= 3;

  // 第二约束：≥1 待确认共识
  const pendingConsensus = state.records.filter(r => r.type === 'consensus' && r.status === 'pending_client_confirm');
  const secondConstraintMet = pendingConsensus.length >= 1;

  // 两个约束都满足才显示红点
  if (firstConstraintMet && secondConstraintMet) {
    elements.candidateBadge.textContent = '🔴';
  }
}
```

**修改文件**: `public/js/app.js`

### Fix 10: `/总结` 指令实现

**PRD 章节**: 3.2 快捷指令 - "/总结"

**问题描述**:
缺少 `/总结` 指令，无法快速查看当前阶段诊断进度。

**修复内容**:
```javascript
function executeSummaryCommand() {
  const stageRecords = state.records.filter(r => r.stage === state.currentStage);
  const facts = stageRecords.filter(r => r.type === 'fact');
  const consensus = stageRecords.filter(r => r.type === 'consensus');
  const confirmed = stageRecords.filter(r => r.status === 'confirmed');

  const summaryLines = [
    `【${state.currentStage}阶段总结】`,
    `已确认事实: ${facts.filter(f => f.status === 'confirmed').length} 条`,
    `待确认判断: ${consensus.filter(c => c.status === 'pending_client_confirm').length} 条`,
    `已达成共识: ${confirmed.filter(c => c.type === 'consensus').length} 条`,
    `完整度: ${Math.round(state.completeness)}%`
  ];

  // 关键发现（前3条）
  if (facts.length > 0) {
    summaryLines.push('', '关键发现:');
    facts.slice(0, 3).forEach((f, i) => {
      summaryLines.push(`  ${i + 1}. ${f.content.slice(0, 50)}...`);
    });
  }

  elements.suggestionQuestion.textContent = summaryLines.join('\n');
}
```

**修改文件**: `public/js/app.js`

### Fix 11: `/记` 默认状态改为 `pending_client_confirm`

**PRD 章节**: 4.2 路径一 - "顾问敲 /确认，状态从 pending_client_confirm 变为 confirmed"

**问题描述**:
`/记` 创建记录时默认状态为 `recorded`，但 PRD 4.2 要求 `/确认` 操作的对象是 `pending_client_confirm` 状态的记录。两者不匹配导致 `/确认` 找不到待确认记录。

**修复内容**:
- `consensusChain.js` `addRecord()` 默认状态从 `recorded` 改为 `pending_client_confirm`
- 前端 `getStatusText()` 映射：`pending_client_confirm` → '待确认'，`recorded` → '已记录'

**状态流转**:
```
/记 → pending_client_confirm（UI: "待确认"）
/确认 → confirmed（UI: "已确认"，同步飞书）
```

**修改文件**: `src/core/consensusChain.js`, `public/js/app.js`

### Fix 12: `/确认` 智能选择逻辑

**PRD 章节**: 4.2 路径一 - "确认选中内容的状态同步到飞书；如果没有选中确认最新一条"

**问题描述**:
原 `/确认` 仅支持确认最新一条 `pending_client_confirm` 记录，不支持优先确认选中的候选记录。

**修复内容**:
- 新增 `POST /api/sessions/:id/confirm` 智能确认端点
- 前端 `executeConfirmCommand()` 优先使用 `state.candidateId`，否则由后端选最新 pending
- `selectCandidate()` 保存选中记录 ID 到 `state.candidateId`

**逻辑优先级**:
1. 有选中候选 → 确认该候选记录
2. 无选中 → 确认最新一条 `pending_client_confirm` 记录

**修改文件**: `server.js`, `public/js/app.js`

### Fix 13: 飞书同步时机（仅确认后同步）

**PRD 章节**: 4.4 - "只有 status=confirmed 的记录才同步到飞书诊断共识表"

**问题描述**:
原实现中所有记录（包括 `pending_client_confirm`）创建后立即同步到飞书，客户侧可见未确认的记录。

**修复内容**:
- `consensusChain.js` `addRecord()` 的 `syncToFeishu` 参数默认改为 `false`
- `confirmRecord()` 内部调用 `_syncToFeishu()`，仅确认时同步
- `server.js` 的 `change` 事件处理器移除重复的飞书同步调用（由 consensusChain 内部处理）

**同步规则**:
| 操作 | 飞书同步 |
|------|---------|
| `/记` (pending_client_confirm) | ❌ 不同步 |
| `/确认` (confirmed) | ✅ 同步一次 |

**修改文件**: `src/core/consensusChain.js`, `server.js`

### Fix 14: Fastify 空 JSON Body 解析

**问题描述**:
前端 `apiRequest` 发送 `POST` 请求时，无 body 的请求携带 `Content-Type: application/json` 头，Fastify 抛出 `FST_ERR_CTP_EMPTY_JSON_BODY` 错误。

**修复内容**:
```javascript
fastify.removeAllContentTypeParsers();
fastify.addContentTypeParser('*', { parseAs: 'string' }, (req, body, done) => {
  if (body && body.trim()) {
    done(null, JSON.parse(body));
  } else {
    done(null, {});
  }
});
```

**修改文件**: `server.js`
