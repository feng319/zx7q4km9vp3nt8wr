## 一、产品定位与核心场景

### 1.1 产品定位

**顾问现场作战系统**(Consultant Field Cockpit):让咨询顾问在客户面前**反应快、判断准、记忆全**的现场作战工具。

**关键隐喻**:顾问是飞行员,系统是仪表盘和副驾驶。客户看到的是飞行员的判断力和操控力,看不到仪表盘细节,但能感受到飞行的稳定和精准。

### 1.2 核心场景锁定

**B为主(会议中实时协作)、D为辅(会前会后延伸)**

- **会议中实时协作**(75%投入):双层架构、候选生成、共识沉淀、成交触发
- **会前准备**(10%投入):生成作战卡,让顾问进会议时有底
- **会后整理**(15%投入):一键生成备忘录,当天发客户

**关键判断**:会前和会后的功能必须**完全服务于会议中**。会前作战卡的目标是"让顾问在会议前30分钟把状态调整到最佳";会后备忘录的目标是"在会议成交势能还没消散前推动客户进入下一步"。

**反向约束**:如果某个会前或会后功能不服务于"成交时刻",直接砍掉,不做。

### 1.3 双层架构设计

**顾问作战层**(Streamlit本地应用):
- 用户:只有顾问看得到
- 功能:候选生成、共识链记录、知识库召回、追问建议
- 特征:信息密度高,调试信息全显,是顾问的"驾驶舱"

**客户共识层**(飞书多维表格):
- 用户:顾问+客户共同可见,会议中投屏给客户
- 字段:
  - **静态字段(9个)**:公司名、显性诉求、产品线、客户群体、收入结构、毛利结构、交付情况、资源分布、战略目标
  - **动态字段(2个)**:当前追问(追问时写入,回答录入后清空)、诊断进度(0-100%,实时更新)
- 功能:字段填充、客户确认、共识沉淀、备忘录生成的数据源
- 特征:UI干净、字段标准、客户能直接修改、所有改动实时同步回Streamlit

**关键原则**:客户**永远只看飞书**,Streamlit对客户隐藏。

### 1.4 会议SOP(操作规范)

**演示模式切换时机**:
- 从飞书投屏切回Streamlit操作时,先切换演示模式(Ctrl+Shift+D或F11),再开始操作
- 主动展示Streamlit给客户时,切换演示模式→全屏→展示≤30秒→切回工作模式

**候选生成调用时机**:
- 感受到"客户在问决策问题"时,敲`/候选`
- 看到`/候选`指令旁出现红点徽标时,说明共识链已满足候选生成条件

**共识链记录习惯**:
- 听到客户陈述事实,立即敲`/记 <内容>`
- 客户确认某个判断后,敲`/确认`锁定

**阶段切换操作**:
- 敲`/切 商业模式`或`/切 行业演示`手动切换阶段
- 阶段切换触发候选预计算重算

## 二、知识召回机制设计

### 2.1 召回策略:C策略(关键词自动+指令补充)

**核心判断**:召回可以激进自动化,因为召回错了代价小(只是"信息没用上"),不会让顾问在客户面前翻车。

**召回机制的输出对象是顾问,不是客户**。所有SKU卡片只在Streamlit顾问端显示,不同步到飞书客户侧。顾问决定是否口播、如何口播,系统不替顾问做这个判断。

**三层召回机制**:

| 层级 | 触发方式 | 召回数量 | 显示位置 |
|---|---|---|---|
| 第一层:指令召回 | 顾问敲`/案例` `/框架` `/对比` | Top 5 | 主对话区,立即弹出 |
| 第二层:关键词自动召回 | 监听对话中的行业关键词 | Top 3 | 侧边栏"备弹区",常驻不打断 |
| 第三层:阶段软排序 | 同等关键词匹配下,当前阶段权重+0.3 | 影响排序,不直接触发 | 与第二层共享UI |

### 2.2 关键词提取机制

**第一步:关键词词典**(50-80个核心概念)
- 从ontology自动抽取,每个概念配3-5个同义词/缩写
- 例:"虚拟电厂"绑定"VPP"、"聚合商"、"负荷聚合"、"调度平台"

**第二步:意图判定**(Day 3增强,可选)
- 用轻量LLM判定"客户是在认真讨论这个概念,还是只是顺嘴一提"
- 耗时0.5-1秒,把误召回率压到10%以下
- **降级方案**:Day 1-2用纯字典匹配+5秒窗口去重

### 2.3 UI设计:前后台分区

**前台主弹区**(主对话流中):
- 只有顾问主动用`/`指令调用时才显示
- 占据视线焦点,立即弹出Top 5 SKU卡片
- 每张卡显示:标题+一句话摘要+可信度标签(🟢🟡🔴)

**后台备弹区**(屏幕右侧200-300px边栏):
- 根据当前对话语境自动召回Top 3 SKU,常驻不打断
- 顾问需要时一眼扫到,用Tab键聚焦展开
- 不需要时就是边栏背景,不打断思路
- **渐进式更新**:新召回内容不直接覆盖,在卡片右上角显示小红点提示"有新备弹"
- **演示模式下**:备弹区整体折叠为小图标,顾问需要时可展开,默认不显示

**SKU弹药卡片与候选方案卡片的视觉区分**:

| 维度 | SKU弹药卡片 | 候选方案卡片 |
|---|---|---|
| 位置 | 屏幕右侧备弹区(边栏) | 主对话区(中央,覆盖式弹出) |
| 颜色 | 灰白底色 | 蓝色边框+浅蓝底色 |
| 形态 | 单卡片,纵向排列 | 2-3张卡横向并列 |
| 标识 | 🟢🟡🔴可信度标签 | 卡顶大字"候选A/B/C" |
| 操作 | Tab键展开摘要 | 数字键1/2/3选中,回车确认 |
| 本质 | 被动信息(看完顾问自己用) | 主动决策(选一个进入共识链) |

### 2.4 限流机制

**召回计算(本地)**:
- 同一关键词5秒内不重复触发
- 备弹区最短刷新间隔30秒
- 单次会议召回调用上限50次(超出后停止自动召回,顾问仍可用`/搜`指令手动触发)

**飞书API写入(远程)**:
- 频控单位:每个API×每个应用×每个租户,独立计算
- 写入类接口:等级5~6(1~5次/秒),保守按等级5(1次/秒)实现
- 读取类接口:等级7~8(10~20次/秒)
- QPM+QPS双重限制:任意一个触发上限即返回429
- 多维表格API不支持提频(硬约束,不可绕过)

**429处理逻辑**:
- 触发后读取响应头`x-ogw-ratelimit-reset`字段
- 等待对应秒数后重试(禁止立即重试)
- 批量写入使用`lark-cli --page-delay 500ms`(保守值)
- 实测稳定后可降至200ms

**自定义机器人(发备忘录场景)**:
- 限制:100次/分钟、5次/秒
- 避开10:00/17:30整点半点时段
- 不支持提频,与多维表格写入共享此约束

### 2.5 降级路径

- 关键词匹配不准→顾问用`/搜 <自定义关键词>`强制召回
- 备弹区频繁刷新→采用渐进式更新(小红点提示)
- 召回的SKU与会议节奏不同步→显示"召回时间戳",3分钟以上的旧备弹自动半透明化

## 三、候选生成器设计(MDU核心)

### 3.1 触发时机:D策略(主动调用为主+红点徽标提示)

**核心判断**:候选的对象是判断,不是事实。候选错了,顾问真的把它讲给客户,就是产品级事故。因此候选生成必须比召回更克制。

**主路径:顾问敲`/候选`主动调用**
- 顾问感受到"客户在问决策问题",敲一个键
- 系统在主对话区**立即**弹出2-3个候选方案卡
- 响应延迟必须<1.5秒,否则会议节奏断掉
- 为保证响应速度,候选生成在后台**预计算**

**安全网:共识链满足条件时静默标红**
- 当共识链积累到候选生成条件时,在屏幕角落的`/候选`指令提示符上加一个红点徽标
- 顾问扫一眼看到红点,知道"现在调用候选效果会很好"
- 没看到红点,敲`/候选`也能用,只是系统会提示"证据不足,候选质量可能较低"
- **红点消失时机**:顾问敲`/候选`并完成选择(或主动关闭)后,红点清除;若顾问忽略红点超过5分钟,红点自动熄灭

### 3.2 候选预计算机制

**后台持续维护"当前最佳候选"缓存**:
- 每当共识链新增一条事实、或备弹区召回新SKU、或对话进入新阶段时,触发后台候选生成
- 生成结果存入缓存,**不主动推送给顾问**
- 顾问敲`/候选`时,系统直接从缓存读取,0.2秒响应

**缓存刷新控制**:
- 最短刷新间隔30秒(30秒内多次共识变动只触发一次重算)
- 只有"实质性增量"才重算(新增产品线、新增痛点、新增决策问题)
- 修正不改变语义的内容不触发
- 单次会议候选生成调用上限20次

**缓存失效条件**:
- 顾问手动修正共识链中的某条事实时,缓存必须立即作废,强制重算
- 防止顾问改完事实、敲`/候选`,系统还在用旧数据生成方案

### 3.3 候选质量三约束

每次生成候选时,系统强制检查三个硬约束,**任何一个不满足都不生成候选**:

**第一约束**:共识链必须有≥3条已确认事实
**第二约束**:至少要有一个"待确认假设"或"客户决策问题"作为候选的目标
**第三约束**:每个候选必须能引用≥1个SKU作为证据,且SKU可信度标签为🟢或🟡
- SKU验证使用**备弹区当前缓存**(不发起新的检索)
- 如果备弹区没有🟢/🟡的SKU,则触发一次快速补充召回(Top 1)
- 补充召回失败则约束不满足,返回反向引导

**三约束未满足时的反向引导**:
- 系统返回明确文案:"当前共识不足以生成高质量候选,建议先追问X/Y/Z三个问题"
- 这个反向引导本身就是产品价值——让顾问知道"还差什么信息才能做决定"
- 产品价值必须显式写出,不能只作为隐含逻辑

### 3.4 候选差异度保证

**Prompt强制要求**:
- 三个候选必须分别对应**不同的战略方向**(重资产vs轻资产、自建vs合作、聚焦vs多元)
- 三个候选必须分别对应**不同的风险偏好**(稳健、平衡、激进)
- 三个候选的描述长度必须接近(差异不超过30%)

**生成后自检**:
- 用embedding计算三个候选描述的两两相似度
- 相似度>0.85的判定为伪差异,自动重生成
- **Fallback策略**:重生成上限2次,第2次仍失败时,系统提示顾问"当前共识信息可能不足以支撑三条差异化方向,建议先追问客户的风险偏好或资源约束",同时展示已生成的最优2条候选(不强行凑第3条)

### 3.5 UI设计:决策仪式感

**候选卡弹出时**:
- 主对话区其他内容半透明化
- 三张候选卡占据视觉中心,横向并列
- 卡片上方大字:"这是基于已确认的X条事实生成的方案,请选择最贴合客户局势的方向"

**候选卡视觉规范**:
- 位置:主对话区中央,覆盖式弹出
- 颜色:蓝色边框+浅蓝底色(与SKU弹药卡的灰白底色形成对比)
- 形态:2-3张卡横向并列
- 标识:卡顶大字"候选A/B/C"
- 操作:数字键1/2/3选中,回车确认进入共识链

**选中后行为**:
- 选中的候选自动进入共识链,标记为"待客户确认"状态
- 其他两张候选**淡出但不消失**——保留"刚才考虑过的备选"作为顾问后续口播的弹药
- 等顾问引导客户表态后再敲`/确认`锁定

**退出路径**:
- 按`Esc`键或敲`/关闭`可以收起候选卡
- 三张卡折叠到屏幕右下角作为"待决策"徽标
- 顾问随时可以重新展开

## 四、共识链记录设计

### 4.1 记录触发:D策略(候选自动进入+手动`/记`补充+AI兜底提示)

**核心判断**:共识链是会后备忘录的唯一数据源,它的完整性直接决定备忘录能不能当天发给客户。

**共识链记录的两类内容**:

**第一类:事实确认**(客户产品线5条、主要客户群是工商业、收入80%来自EPC)
- 特征:散落在会议全程、随时出现、没有固定时机
- 记录方式:顾问手动`/记`为主

**第二类:判断确认**(客户认可"产品线过于分散是核心问题"、客户选择了"聚焦储能主航道"的候选方向)
- 特征:有明确的确认时刻、通常在候选选中之后、是成交的直接依据
- 记录方式:候选选中自动进入

**修改已记录内容的处理**:
- 对已记录条目的修改不覆盖原记录
- 新增一条`source=manual_correction`、`replaces=cc_xxx`的记录
- 原记录保留但标记为`superseded`
- 备忘录生成时能追溯"这条判断是被修正过的",也防止飞书侧出现脏数据

### 4.2 三条记录路径

**路径一:候选选中自动进入**(最高优先级,零操作成本)
- 顾问用数字键选中某个候选方案时,该候选**立即自动写入共识链**
- 状态标记为`pending_client_confirm`(待客户确认)
- 同时做三件事:
  1. 共识链新增一条`type=consensus`的记录
  2. 飞书多维表格对应字段实时更新
  3. 四栏状态板的"待确认假设"区出现这条判断
- 顾问之后敲`/确认`,状态从`pending_client_confirm`变为`confirmed`

**路径二:顾问手动`/记`补充**(中优先级,事实类内容主力)
- 格式:`/记 <任意自然语言>`
- 系统自动判断`type`(包含"客户认可"/"我们决定"等词的判定为`consensus`,其余默认为`fact`)
- **Type判断失败时的兜底**:系统在侧边栏绿色闪动后,在共识链区域用灰色小字追加一行"这条记录被归类为fact,如需改为consensus请敲`/改类`"
- 响应:侧边栏共识链区域**闪一下绿色**(0.3秒)表示已记录,无弹窗、无确认、无打断
- 快捷键绑定:`Ctrl+Enter`(或`Cmd+Enter`)

**路径三:AI提示"建议记录X"**(最低优先级,会议后半段兜底)
- **不弹确认窗口,只在备弹区角落显示一行灰色小字**
- 旁边有一个`+`按钮,顾问需要时点一下,不需要时忽略
- **只在以下条件同时满足时才启动**:
  1. 会议已进行20分钟以上
  2. 过去5分钟内共识链没有新增任何记录
  3. 对话中出现了明显的"客户陈述事实"模式:连续≥2句包含数字、比例、产品名称的陈述句,且没有疑问句打断

### 4.3 共识链数据结构

```python
{
  "id": "cc_001",
  "timestamp": "14:23:05",
  "type": "fact" | "consensus",
  "stage": "战略梳理" | "商业模式" | "行业演示",
  "content": "客户产品线5条:工商业储能/户用光伏/VPP/EPC/设备",
  "source": "manual" | "candidate_selected" | "ai_suggested" | "manual_correction",
  "evidence_sku": ["sku_037", "sku_012"],  # 仅consensus类有
  "status": "recorded" | "pending_client_confirm" | "confirmed" | "superseded",
  "confidence": "high" | "medium" | "low",  # 仅ai_suggested有
  "replaces": "cc_xxx",  # 仅manual_correction有
  "superseded_by": "cc_xxx",  # 被替代时填入
  "feishu_record_id": "rec_xxx"  # 飞书多维表格对应记录ID
}
```

**关键字段说明**:
- `source`:记录这条共识是怎么来的。备忘录生成时,`source=candidate_selected`的条目优先级最高
- `status`:区分"已记录"和"已确认"。备忘录的"诊断结论"章节只用`status=confirmed`的条目
- `confidence`:仅`source=ai_suggested`的记录有此字段。备忘录生成时,`confidence=low`的AI建议记录即使`status=recorded`也不进入备忘录正文,只进入附录
- `feishu_record_id`:飞书同步的锚点

**阶段(stage)字段赋值逻辑**:
- 阶段切换由顾问手动触发,敲`/切 商业模式`或`/切 行业演示`
- 系统不自动判断阶段切换
- 当前阶段保存在`st.session_state.current_stage`
- 每条共识链记录写入时,自动从`current_stage`读取当前阶段

### 4.4 飞书侧显示设计

**两张数据表**:

**第一张"客户档案"**:9字段client_profile,会议中实时填充,客户可以直接修改

**第二张"诊断共识"**:
- 每条`status=confirmed`的共识链记录显示为一行
- 列名:"发现内容"、"确认时间"、"建议方向"
- `status=pending`的记录**不显示给客户**,只在Streamlit顾问端可见
- "建议方向"列数据来源:`type=consensus`的记录有`recommendation`字段,内容是候选卡的标题(如"聚焦储能主航道")

**隐藏机制**:客户看到的飞书表格里只有"我们已经达成共识的内容",干净、专业、有说服力;顾问端看到的是完整记录,包括"待确认假设"和"AI提示但未确认"的内容。

### 4.5 备忘录自动生成的映射逻辑

| 共识链内容 | 备忘录章节 |
|---|---|
| `type=fact` + `status=confirmed` | 二、关键发现 |
| `type=consensus` + `status=confirmed` + `source=candidate_selected` | 三、初步建议方向(标注"系统生成") |
| `type=consensus` + `status=confirmed` + `source=manual` | 三、初步建议方向(标注"顾问判断") |
| `type=consensus` + `status=pending_client_confirm` | 四、需要进一步访谈的内容 |
| `type=fact` + `source=ai_suggested` + `status=recorded` + `confidence=high/medium` | 四、需要进一步访谈的内容(标注"待核实") |
| `type=fact` + `source=ai_suggested` + `confidence=low` | 附录(不进正文) |
| client_profile的"显性诉求"字段 | 一、问题重构(原始诉求) |
| 共识链中最早的`type=consensus` + `status=confirmed` | 一、问题重构(诊断后核心问题) |

**关键原则**:这个映射是备忘录生成器的核心逻辑,不需要LLM做任何"创作",只是把结构化数据填入模板。LLM只负责把每条记录的内容"润色成一句通顺的中文",不负责内容本身。

## 五、演示模式设计

### 5.1 设计原则:三级敏感度分区

把Streamlit界面分成三个区域,每个区域的切换行为不同:

**第一级:完全隐藏区**(切换时`display: none`)
- token计数、API响应时间
- SKU编号(`sku_037`格式)
- 可信度标签(🟢🟡🔴+技术说明文字)
- 调试日志、模型名称
- 知识库版本号(工作模式下的调试格式)
- 快捷指令提示栏(`/案例`、`/记`等)

**第二级:内容替换区**(切换时替换显示文本,不隐藏容器)

| 工作模式显示 | 演示模式显示 |
|---|---|
| `[新能源-sku_037] 设备商转运营商路径` | `标杆案例:设备商转运营商路径` |
| `当前阶段:战略梳理 Phase 1/3` | `诊断进度:战略方向梳理` |
| `共识链:7/9 字段已确认` | `诊断完成度:78%` |
| `候选方案(预计算缓存命中)` | `初步建议方向` |
| `候选已进入共识链,状态:待客户确认` | `方向已记录` |

**第三级:保留区**(切换时完全不变)
- 对话主流区的顾问输入框
- 候选方案卡片(演示模式下更清晰,但内容不变)
- 四栏状态板(已确认事实、待确认假设、当前阶段、下一步建议)
- 共识链已确认条目(`status=confirmed`)

### 5.2 切换机制:瞬间完成

**切换必须是0.1秒内完成的离散状态跳变**,不是连续动画。

**技术实现**:
- 通过`st.session_state.demo_mode = True/False` + CSS类切换
- 不重新渲染组件,只改CSS可见性
- Streamlit里0.1秒内完成
- `demo_mode`状态必须持久化到`st.session_state`,并且在每次重渲染时**优先恢复**CSS类

**切换触发方式**:
- 快捷键绑定到`F11`和`Ctrl+Shift+D`(**两个都要实现**)
  - `F11`是全屏切换的肌肉记忆,顾问可在0.5秒内完成"切换演示模式+全屏"两个动作
  - `Ctrl+Shift+D`是有语义的快捷键,不易与其他操作冲突
- **风险点**:Streamlit在某些操作后会触发全量重渲染,全量重渲染会重置CSS状态,导致演示模式被意外退出。必须在Day 2验收前专项测试

**状态徽标**:
- 屏幕右上角显示极小的状态徽标
- 工作模式:灰色小圆点
- 演示模式:蓝色小圆点
- 顾问扫一眼就知道当前模式,不需要读任何文字

### 5.3 主动加分设计

演示模式不只是"隐藏坏东西",还要"主动显示好东西":

**四栏状态板字体放大120%**
- 工作模式:辅助信息,字体可以小
- 演示模式:客户可能瞥一眼,字体必须够大让他们能读懂
- 客户看到"已确认事实:产品线5条/主要客户群:工商业",感受到"顾问在认真记录我说的话"

**知识库版本号从角落移到显眼位置**
- 工作模式:调试格式(`v2.0 · 364 SKUs · last updated 2025-04-30`),归入第一级隐藏
- 演示模式:营销格式,显示为:"**知识库v2.0 · 364个新能源行业案例 · 35条跨域洞察**",归入第二级内容替换,位置从角落移到显眼位置
- 客户瞄一眼,感受到"这是一个有深度积累的专业系统"

**候选方案卡片标题字体加粗加大**
- 工作模式:给顾问快速扫读,信息密度高
- 演示模式:顾问要讲给客户听,视觉层次必须清晰
- 让客户在顾问口播时能跟着看懂

### 5.4 边界问题:三种场景

**场景一:客户看飞书多维表格**(顾问把飞书投屏给客户)
- Streamlit在顾问的笔记本屏幕上,客户看不到
- **演示模式不需要开启**,顾问可以放心用工作模式

**场景二:客户瞥到顾问的笔记本屏幕**(Streamlit在顾问的屏幕上)
- 演示模式必须开启

**场景三:顾问主动将Streamlit界面展示给客户**
- 操作流程:切换演示模式→全屏(F11)→将屏幕转向客户
- 展示内容建议:只展示四栏状态板+共识链已确认条目,不展示候选卡生成过程
- 展示时长建议:≤30秒,展示完毕后切回工作模式继续操作

**操作流程**:
- 顾问从飞书投屏切回自己操作Streamlit时,先切换演示模式,再开始操作
- 这个操作习惯需要顾问在Day 3的演练中刻意练习,直到形成肌肉记忆
- 把这个操作流程写进"会议SOP"里,贴在顾问的笔记本旁边

### 5.5 验收标准

**Day 2下午的关键验收标准**:演示模式截图看起来像正式产品。

**验证方法**:
- 把演示模式截图和工作模式截图并排放
- 如果能一眼看出差异、演示模式明显更专业,说明设计成功
- 如果两张截图差不多,说明"主动加分"设计没做到位,需要回来打磨

**过程验收指标**:
- 截图盲测:把演示模式截图发给一个没有参与过这个项目的朋友,问他"这是什么工具"——如果他的回答里包含"专业"、"系统"、"咨询"等词,说明视觉设计达标;如果他说"这是个聊天机器人",说明演示模式还不够
- 切换速度测试:从顾问感受到"需要切换"到切换完成,全程不超过3秒(包括按键+等待渲染+确认状态徽标变色)

**最终验证指标**:
- Day 3客户演练结束后,客户主动说"你用的这个系统看起来很专业"
- 注意:不是顾问介绍系统,而是客户主动说出来
- 如果需要顾问解释才能让客户觉得专业,说明演示模式的视觉设计还不够

## 六、会前作战卡设计

### 6.1 设计原则

作战卡不是知识库的压缩版,也不是client_profile的打印版。它只回答一个问题:

> **顾问进会议室前10分钟读完,能知道"这场会议要验证什么、遇到什么情况说什么、追问无从下手时往哪走"。**

**三条硬约束**:

1. **1页A4,不能超** - 超过一页的作战卡在会议中几乎不会被翻到第二页
2. **内容是台词,不是信息** - 顾问需要台词,不是原材料
3. **诊断假设是灵魂** - 所有问题、案例、话术都围绕核心假设展开

### 6.2 双模式自动切换

根据client_profile完整度自动切换:

| 条件 | 模式 | 会议目标 |
|---|---|---|
| 完整度≥60% | **验证假设版** | 深挖已知信息,验证/推翻诊断假设 |
| 完整度<60% | **信息建立版** | 系统性追问,把9个字段从零建立起来 |

**边界说明**:作战卡是会前生成的静态文档,会议中不切换模式。模式切换只在**下一次生成作战卡时**生效。会议中完整度跨越60%的信号,通过6.5节的三栏布局状态切换来传递,而不是重新生成作战卡。

**完整度计算规则**:
- 9个字段等权重,每个字段非空即计11%(9个字段满分99%)
- 第100%由至少1条`status=confirmed`的共识链记录触发
- "部分填充"(橙色)判定:字段有内容但内容少于10个字符,或顾问主动标记为"待核实"

- 预设追问树和动态追问建议形成互补,而不是两套平行系统

### 6.5 会议中的追问交互设计

**主界面三栏布局**(比例3:5:2):

```
┌─────────────────┬───────────────────────────┬──────────────┐
│   左栏(3)       │      中栏(5)               │  右栏(2)     │
│                 │                           │              │
│  字段完整度      │    主对话/追问输入区        │  当前追问     │
│  实时状态板      │                           │  建议卡       │
│                 │  顾问输入+AI响应流          │              │
│  ✓ 已确认(绿)   │                           │  下一个最该   │
│  ◑ 部分填充(橙) │                           │  追问的问题   │
│  ○ 未填充(灰)   │                           │  +分支选项   │
│                 │                           │              │
└─────────────────┴───────────────────────────┴──────────────┘
```

**右栏追问建议卡**:
- 只显示**一个问题**,不显示问题列表
- 三个按钮:`[用这个问题]` `[跳过,下一个]` `[我来写]`
- 分支追问时切换为分支选项模式(最多3个选项,键盘A/B/C快速选择)

**追问的连续闭环**:
1. 顾问点击`[用这个问题]`
2. 中栏填入问题文本
3. 顾问口播给客户
4. 顾问输入客户回答
5. 系统自动捕获→字段映射
6. 左栏实时刷新进度条跳动
7. 飞书字段同步更新
8. 右栏推送下一追问建议

**完整度60%的状态切换**:
- 左栏进度条变绿,显示"基础诊断信息已就绪"
- 右栏从"追问建议"切换为"候选生成就绪",提示"输入`/候选`触发初步诊断方向"
- 中栏顶部出现提示:"已收集足够信息进入诊断阶段,也可继续深挖"
- **防抖逻辑**:完整度跨越60%的状态切换有**30秒冷却期**,冷却期内不再触发反向切换

### 6.6 飞书集成方案

**核心决策:Streamlit主导,飞书只做展示**

飞书侧在9个静态字段之外,增加两个动态字段:

**`当前追问`字段**(单行文本):
- 顾问点击`[用这个问题]`时自动写入问题文本
- 客户回答录入后自动清空
- 客户投屏看到这个字段,知道当前在讨论什么
- **写入防抖**:顾问点击"用这个问题"后1秒内只触发一次写入,不允许连续写入;客户回答录入后的清空操作合并到下一次写入

**`诊断进度`字段**(百分比):
- 实时显示9个字段的完整度
- 客户看到这个数字在增长,感受到"我们的诊断在往前走"

**飞书投屏的客户体验**:
- 动态区(顶部):`当前追问`+`诊断进度`
- 静态区(下方):9个字段的已确认内容
- 客户在投屏上看到问题出现、字段被填充、进度条增长——传递"这是一套有方法论支撑的系统性诊断"的信号

### 6.7 生成逻辑

**第一步:规则过滤**(0.1秒,不依赖AI)
- 根据client_profile的行业角色×目标场景×痛点标签
- 从知识库硬过滤出相关SKU候选集(通常20-35个)

**第二步:优先级排序**(纯数值计算,不依赖AI)
- 按"角色匹配度0.4+痛点匹配度0.3+SKU可信度0.2+当前阶段相关性0.1"加权排序
- 取Top 15

**第三步:AI语言润色**
- 把Top 15的SKU摘要传给LLM
- **按作战卡区块分批传入**:战略问题区块传Top 3 SKU,商业模式区块传Top 3 SKU,备弹区块传Top 3 SKU,分三次调用LLM,每次输入控制在3条摘要以内
- 做一件事:**把SKU摘要转化成顾问可以直接口播的台词**
- AI不做内容取舍,只做格式转换

**完整度判断前置**:在三步之前,先计算client_profile完整度,决定生成验证假设版还是信息建立版。

### 6.8 输出格式

- 统一输出Word格式(`.docx`),不用PDF
- 顾问可以在会议前最后5分钟手动微调
- 字体规范:标题14pt加粗,区块标题11pt加粗,正文10pt,行间距1.2
- **字段映射说明**:作战卡里的内容和client_profile字段的映射关系是单向的——作战卡是从client_profile生成的,但如果顾问在Word里手动修改了作战卡内容,这个修改**不回写到client_profile**

### 6.9 验收标准

**指标一**:Day 3客户演练时,顾问把作战卡打印出来放在桌上,会议中至少翻看了3次

**指标二**:顾问第一次使用系统,在没有培训的情况下,30分钟内完成9个字段中至少6个字段的填充,且每两个相邻字段之间的间隔不超过4分钟(即顾问不会在某个字段上卡超过4分钟)。这个标准可以从系统日志里直接读出来,不依赖主观观察。

## 七、备忘录生成设计

### 7.1 设计原则:一键生成即可发送

**核心目标**:会议结束2分钟内生成,当天发给客户,顾问无需修改即可发送。

**关键约束**:
- 生成时间<30秒
- 顾问只需要做一件事:确认发送
- 从"会议结束"到"客户收到备忘录"全流程控制在2分钟内

### 7.2 三层架构

```
第一层:数据提取层(确定性规则,零AI)
    ↓
第二层:结构组装层(模板+规则,零AI)
    ↓
第三层:语言润色层(AI,但有严格约束)
```

**关键原则**:越靠近"内容"的层,越不依赖AI;越靠近"语言"的层,AI介入越多。AI只被允许改变表达方式,不被允许改变内容本身。

**架构原则**:三层是**单向流水线**,不允许第三层的输出反向影响第一、二层的数据。LLM润色后的文字不回写共识链,也不修改client_profile。

### 7.3 第一层:数据提取(确定性规则)

从共识链和client_profile里,按固定规则提取每个章节需要的原始数据。完全不依赖AI,纯Python逻辑。

| 备忘录章节 | 数据来源 | 提取规则 |
|---|---|---|
| 一、问题重构 | client_profile.显性诉求+共识链中最早的`type=consensus` `status=confirmed` | 显性诉求作为"原始诉求",第一条确认共识作为"诊断后核心问题" |
| 二、关键发现 | 共识链中所有`type=fact` `status=confirmed` | 按阶段分组(战略/商业模式/行业),每组最多3条,按优先级排序:`source=candidate_selected`>时间戳最早>时间戳最新 |
| 三、初步建议方向 | 共识链中所有`type=consensus` `status=confirmed`且(`source=candidate_selected`或`source=manual`) | 只取候选选中或手动确认的条目,保留原始候选描述+证据SKU |
| 四、需要进一步访谈 | 共识链中所有`status=pending_client_confirm`+未填充字段+`source=ai_suggested` `confidence=high/medium` | pending条目作为"待确认判断",未填字段作为"待收集信息",AI建议标注"待核实" |
| 附录 | `source=ai_suggested` `confidence=low` | 低置信度AI建议,不进正文 |
| 五、建议下一步合作 | 固定模板+服务包定价 | 根据诊断深度自动推荐服务包档位 |

### 7.4 第二层:结构组装(模板+规则)

把第一层提取的数据,按章节模板组装成结构化的"要点列表"。输出不是最终文字,是"有结构的原材料"。

**示例:"二、关键发现"的结构**

```python
{
  "chapter": "二、关键发现",
  "sections": [
    {
      "label": "战略层面",
      "facts": [
        "产品线共5条:工商业储能/户用光伏/VPP/EPC/设备供应",
        "EPC工程收入占比70%,是主要现金流来源",
        "各产品线资源投入分散,缺乏明确主航道"
      ]
    },
    {
      "label": "商业模式层面",
      "facts": [
        "主要收入为一次性工程款,缺乏持续性收入",
        "客户复购率低,主要依赖新客户获取"
      ]
    }
  ]
}
```

**关键设计**:
- `facts`里的每条内容来自共识链的原始记录,**一字不改**
- 每个section最多3条facts,硬限制
- 超过3条的按优先级降级:`source=candidate_selected`>时间戳最早>时间戳最新,其余降级到"需要进一步访谈"章节

### 7.5 第三层:语言润色(AI,严格约束)

把第二层的结构化要点传给LLM,做且只做一件事:**把要点列表转化成连贯的段落文字**。

**五条硬约束**:

**约束一:禁止添加新内容**
- LLM只能润色已有要点,不能补充任何要点里没有的信息
- Prompt明确写:"你只能用下面的要点,不能添加任何额外信息"

**约束二:字数上限**
- 关键发现章节不超过200字
- 建议方向章节不超过150字
- 超出上限自动截断

**约束三:保留原始数据**
- 数字、比例、产品名称、SKU引用必须原样保留
- Prompt明确写:"所有数字和专有名词必须与原始要点完全一致"

**约束四:语气规范**
- 备忘录是B端商务文档,语气必须是"专业、客观、建设性"
- 不能出现"非常"、"极其"、"完全"等夸张词汇

**约束五:润色失败时的降级**
- 当LLM润色超时或返回错误时,系统必须直接使用第二层的结构化要点列表作为该章节的输出
- 不润色,直接展示要点
- 不跳过章节或显示空白

**润色的本质**:"连接词+逻辑过渡",不是"重新写作"。AI做的事情是把孤立事实用逻辑关系连接起来,内容不变。

### 7.6 第五章节特殊设计(服务包嵌入)

"五、建议下一步合作方式"不走三层流程,而是**固定模板+动态推荐**:

```
五、建议下一步合作方式

本备忘录基于[时长]分钟初步诊断生成,
覆盖了战略梳理、商业模式、行业演示三个维度的初步判断。

基于本次诊断深度,建议进入:

□ [推荐档位]·[服务名称]([价格])
  适合:[一句话说明为什么推荐这个档位]

□ [次选档位]·[服务名称]([价格])
  适合:[一句话说明适用场景]

请在下方勾选您的意向,或直接回复本消息:
□ 希望进入下一阶段合作
□ 需要内部讨论后回复
□ 建议先了解类似案例
```

**推荐档位自动判断逻辑**:
- 共识链确认条数≥5且完整度≥70%,推荐"商业模式专项咨询(1999)"
- 共识链确认条数3-4且完整度50-70%,推荐"战略主线确认工作坊(19800)"
- 完整度<50%,推荐"初步诊断深化(599)"作为入门档
- **兜底规则**:以上三条均不满足时,默认推荐中间档(战略主线确认工作坊19800)

**规则驱动,不是AI判断,结果稳定可预期。**

**价格显示规则**:
- 价格字段从系统配置文件读取,不硬编码在生成逻辑里
- 顾问可以在会议结束后、发送前的2分钟窗口内修改价格
- 这是"顾问确认发送"这一步里允许的唯一修改项
- 价格是商务谈判的变量,不能被系统锁死

### 7.7 备忘录完整结构

```
客户初步诊断备忘录
[客户名]·[日期]·仅供[客户公司名]内部参考

一、问题重构
原始诉求:[client_profile.显性诉求]
诊断后的核心问题:[第一条确认共识,AI润色]

二、关键发现
[战略层面:2-3条,AI连接成段落]
[商业模式层面:1-2条,AI连接成段落]
[行业背景层面:1-2条,AI连接成段落]

三、初步建议方向
方向A:[候选方案A,保留原始描述,AI补充一句逻辑说明]
方向B:[候选方案B,同上]
(如有方向C则列出)
注:来源标注(系统生成/顾问判断)在顾问预览界面显示,最终Word文件中自动去除

四、需要进一步访谈的内容
[pending条目按部门分组]
[未填充字段转化为"建议了解:XXX"]
[AI建议条目标注"待核实"]

[固定模板+动态服务包推荐+意向勾选框]

---
本备忘录基于[时长]分钟初步诊断生成

附录:补充参考(仅供内部参考,不影响上述建议)
[confidence=low的AI建议]
```

**页脚两行字**:专业感的背书,也是"这是初步诊断,后续有深化空间"的隐性信号。

**"仅供[客户公司名]内部参考"**:语义精准——既传递了专业感,又明确了"可以在公司内部传阅",不会阻断内部传播。

### 7.8 工程保证

**生成时间<30秒**:
- 第一层和第二层纯Python,<0.1秒
- 第三层5次LLM调用(每章节一次),使用**信号量控制,最大并发数3**,第四章和第五章串行等待前三章完成后再调用,总时间<15秒
- Word文件生成,<5秒
- 总时间<30秒

**顾问只需确认发送**:
- 不需要修改内容,不需要调整格式
- 只需要看一眼"这份备忘录是否代表今天会议的结论"
- 然后点"发送给客户"

**lark-cli一键发送**:
- 备忘录生成后,Streamlit显示"发送给客户"按钮
- 点击后调用lark-cli把Word文件通过飞书消息发给客户
- 从"会议结束"到"客户收到备忘录"全流程<2分钟

### 7.9 验收标准

**Day 3客户演练结束,顾问看完生成的备忘录说"这就是我想发给客户的"**:
- 注意:是"就是",不是"差不多",不是"改改就行"
- 如果顾问说"还需要改很多",问题不在备忘录生成器,而在共识链的记录质量

## 十一、Day 3 实施详细设计

### 11.1 会前作战卡模块设计

#### 11.1.1 模块结构

```python
# src/core/battle_card_generator.py
"""会前作战卡生成器"""
from dataclasses import dataclass
from typing import Optional, List
from docx import Document
from io import BytesIO
from datetime import datetime

@dataclass
class BattleCard:
    """作战卡数据结构"""
    company: str
    date: str
    consultant: str
    mode: str  # "hypothesis" | "info_building"
    completeness: float
    content: dict  # 各区块内容

class InsufficientSkuError(Exception):
    """SKU数量不足异常"""
    pass

class BattleCardGenerator:
    """作战卡生成器"""

    def __init__(self, feishu_client, llm_client):
        self.feishu_client = feishu_client
        self.llm_client = llm_client
        # 复用 FeishuClient.calc_completeness，不单独实现

    def generate(self, company: str, consultant: str = "") -> BattleCard:
        """生成会前作战卡

        Args:
            company: 客户公司名称
            consultant: 顾问姓名（可选，由调用方从 session_state 或 feishu_client 获取）
        """
        # 1. 获取客户档案
        profile = self.feishu_client.get_client_profile(company)

        # 2. 计算完整度（复用 FeishuClient）
        completeness = self.feishu_client.calc_completeness(profile)

        # 3. 根据完整度选择模式
        if completeness >= 0.6:
            content = self._generate_hypothesis_version(profile)
            mode = "hypothesis"
        else:
            content = self._generate_info_building_version(profile)
            mode = "info_building"

        return BattleCard(
            company=company,
            date=datetime.now().strftime("%Y-%m-%d"),
            consultant=consultant,  # 由调用方传入，或留空
            mode=mode,
            completeness=completeness,
            content=content
        )

    def _generate_hypothesis_version(self, profile: dict) -> dict:
        """生成验证假设版（完整度≥60%）"""
        # 数据流：规则过滤 → 优先级排序 → LLM润色(含约束prompt) → 硬约束校验

        # 1. 规则过滤：根据角色×场景×痛点过滤SKU
        candidate_skus = self._filter_skus(profile)

        # 2. 优先级排序：加权计算Top 15
        top_skus = self._rank_skus(candidate_skus, profile)

        # 3. SKU最小数量保护：确保至少有6条用于核心区块
        MIN_SKU_COUNT = 6
        if len(top_skus) < MIN_SKU_COUNT:
            raise InsufficientSkuError(
                f"SKU召回不足{MIN_SKU_COUNT}条，当前{len(top_skus)}条，无法生成高质量作战卡"
            )

        # 4. LLM润色：分批传入，转化为口播台词
        #    Prompt包含五条硬约束（复用 memo_generator.polish_chapter）
        #    切片安全：Python切片越界不报错，但可能返回空列表
        content = {
            "diagnosis_hypothesis": self._generate_hypothesis(top_skus[:3], profile),
            "strategy_questions": self._generate_questions(top_skus[3:6], "战略梳理"),
            "business_questions": self._generate_questions(top_skus[6:9], "商业模式"),
            "demo_scripts": self._generate_scripts(top_skus[9:12] if len(top_skus) >= 12 else []),
            "risk_responses": self._generate_risk_responses()
        }

        # 5. 硬约束校验：确保诊断假设引用≥1个🟢/🟡SKU
        self._validate_constraints(content, top_skus)

        return content

    def _generate_info_building_version(self, profile: dict) -> dict:
        """生成信息建立版（完整度<60%）"""
        # 追问树区分两种模式：
        # - 预设追问树：固定分支，作战卡静态显示
        # - 动态追问建议：右栏实时生成，顾问点击"我来写"触发

        content = {
            "missing_fields": self._identify_missing_fields(profile),
            "strategy_tree": PRESET_STRATEGY_TREE,  # 预设固定分支
            "business_tree": PRESET_BUSINESS_TREE,
            "demo_scripts": self._generate_scripts(self._get_top_skus(profile)[:3]),
            "risk_responses": self._generate_risk_responses()
        }

        return content

    def _render_to_word(self, battle_card: BattleCard) -> bytes:
        """渲染为Word文档

        Returns:
            bytes: Word文档的字节流（用于下载或发送）
        """
        doc = Document()

        # 标题区
        title = doc.add_paragraph()
        mode_text = '验证假设版' if battle_card.mode == 'hypothesis' else '信息建立版'
```

#### 11.2.2 模块结构

```python
# src/core/feishu_sync.py
"""飞书实时同步模块"""
import threading
import time
import queue
import json
from typing import Callable, Dict, Any, List, Optional

class FeishuSync:
    """飞书实时同步（轮询方案）"""

    def __init__(self, feishu_client, on_record_change: Callable, poll_interval: int = 30):
        """
        Args:
            feishu_client: 飞书客户端
            on_record_change: 变更回调函数
            poll_interval: 轮询间隔（秒），默认30秒
        """
        self.feishu_client = feishu_client
        self.on_record_change = on_record_change
        self.poll_interval = poll_interval

        # 线程安全队列：后台线程只写队列，不直接写 session_state
        self.change_queue = queue.Queue()

        self._last_snapshot: Dict[str, str] = {}  # record_id -> JSON序列化后的字符串
        self._running = False
        self._thread: Optional[threading.Thread] = None
        self._lock = threading.Lock()  # 保护 _running 和 _thread 的并发访问

    def start_listening(self):
        """启动后台轮询线程"""
        with self._lock:
            if self._running:
                return
            if self._thread is not None and self._thread.is_alive():
                # 旧线程还在运行，等待其结束
                self._running = False
                self._thread.join(timeout=5)

            self._running = True
            self._thread = threading.Thread(target=self._poll_loop, daemon=True)
            self._thread.start()

    def stop_listening(self):
        """停止轮询线程"""
        with self._lock:
            self._running = False
            thread_to_join = self._thread

        if thread_to_join is not None:
            thread_to_join.join(timeout=5)

    def _poll_loop(self):
        """轮询主循环"""
        while self._running:
            self._check_changes()
            time.sleep(self.poll_interval)

    def _check_changes(self):
        """检查变更（合并 handler）"""
        try:
            records = self.feishu_client.list_records()
            for record in records:
                rid = record.get("record_id")
                if not rid:
                    continue

                # 使用 JSON 序列化进行比较，避免 datetime 等类型的误判
                record_json = json.dumps(record, sort_keys=True, default=str)
                if self._last_snapshot.get(rid) != record_json:
                    # 后台线程只写队列，不直接写 session_state
                    self.change_queue.put({
                        "record_id": rid,
                        "data": record,
                        "change_type": "update"
                    })
                    self._last_snapshot[rid] = record_json
        except Exception as e:
            # 异常写入队列，主线程处理
            self.change_queue.put({
                "change_type": "error",
                "error": str(e)
            })

    def get_pending_changes(self) -> List[Dict]:
        """获取待处理的变更记录（主线程调用）"""
        changes = []
        while not self.change_queue.empty():
            changes.append(self.change_queue.get())
        return changes

    def process_changes_in_main_thread(self, session_state):
        """在主线程中处理变更（Streamlit回调）"""
        changes = self.get_pending_changes()
        for change in changes:
            if change.get("change_type") == "error":
                # 错误处理
                session_state.last_sync_error = change["error"]
            else:
                # 正常变更处理
                self.on_record_change(change["data"], session_state)
```

```python
# src/core/fallback_handler.py
"""降级处理器"""
from typing import Optional, Callable
from dataclasses import dataclass
from enum import Enum

class FallbackType(Enum):
    """降级类型"""
    FEISHU_API = "feishu_api"
    LARK_CLI = "lark_cli"
    LLM_TIMEOUT = "llm_timeout"
    KNOWLEDGE_RECALL = "knowledge_recall"

@dataclass
class FallbackResult:
    """降级结果"""
    success: bool
    fallback_type: FallbackType
    message: str
    data: Optional[dict] = None

class FallbackHandler:
    """统一降级处理器"""

    def __init__(self):
        self.fallback_counts = {t: 0 for t in FallbackType}

    def handle_feishu_failure(self, operation: str, error: Exception) -> FallbackResult:
        """处理飞书API失败"""
        self.fallback_counts[FallbackType.FEISHU_API] += 1

        return FallbackResult(
            success=True,
            fallback_type=FallbackType.FEISHU_API,
            message=f"飞书同步失败，已保存到本地。会议结束后请手动同步。错误：{error}",
            data={"operation": operation, "local_cached": True}
        )

    def handle_llm_timeout(self, generator: Callable, timeout_seconds: int = 10) -> FallbackResult:
        """处理LLM响应超时"""
        self.fallback_counts[FallbackType.LLM_TIMEOUT] += 1

        try:
            result = generator()
            return FallbackResult(
                success=True,
                fallback_type=FallbackType.LLM_TIMEOUT,
                message="使用同步生成模式",
                data=result
            )
        except Exception as e:
            return FallbackResult(
                success=False,
                fallback_type=FallbackType.LLM_TIMEOUT,
                message=f"LLM生成失败：{e}"
            )

    def handle_knowledge_recall_failure(self, manual_query: str, knowledge_base) -> FallbackResult:
        """处理知识库召回失败"""
        self.fallback_counts[FallbackType.KNOWLEDGE_RECALL] += 1

        results = knowledge_base.search(manual_query, top_k=5)
        return FallbackResult(
            success=len(results) > 0,
            fallback_type=FallbackType.KNOWLEDGE_RECALL,
            message=f"手动召回结果：{len(results)}条",
            data={"results": results, "query": manual_query}
        )

    def get_fallback_report(self) -> dict:
        """获取降级统计报告"""
        return {
            "total_fallbacks": sum(self.fallback_counts.values()),
            "by_type": self.fallback_counts
        }
```
