## **第一部分：项目概述**

### **一、改造背景**

当前系统基于 Python 3.12 + Streamlit 实现，三天 Sprint 已经完成全部核心功能。但在演练准备阶段暴露出三类问题，影响系统的稳定性和可迁移性。

第一类是**环境一致性问题**。Python 版本路径冲突、依赖隔离不彻底、`encodings` 模块加载失败等错误在不同设备上反复出现，每次切换环境都需要排查。第二类是**框架架构限制**。Streamlit 的全量重渲染机制不适合需要实时响应的会议场景，演示模式切换、候选卡弹出、共识链实时刷新都需要特殊处理来规避重渲染副作用。第三类是**部署交付门槛**。当前方案需要顾问端安装 Python、配置环境变量、记忆启动命令，对非技术用户不友好，长期推广困难。

迁移至 Node.js 的核心动机不是性能提升，而是**降低环境复杂度和提升交付体验**——开发者熟悉 Node.js 生态，启动只需一个命令，未来打包为桌面应用或部署上线都更顺畅。

### **二、改造目标**

改造完成后，系统应在三个维度上有明确提升。

**环境部署维度**：从"安装 Python 3.12 + 配置路径 + 安装依赖 + 配置 lark-cli + 启动 Streamlit"五步流程，简化为"`npm install` + `npm start`"两步流程。新设备迁移时间从 30 分钟降至 5 分钟以内。

**用户体验维度**：消除 Streamlit 重渲染导致的演示模式意外退出、候选卡闪烁、共识链记录延迟刷新等问题。WebSocket 实时推送替代轮询，候选卡弹出响应时间从 1.5 秒降至 0.5 秒以内。

**长期演进维度**：技术栈与未来计划保持一致——Electron 桌面应用打包、云端多用户部署、移动端适配，都基于 Node.js 生态有成熟方案，不需要二次重写。

### **三、技术方案对比**

在确定 Node.js 之前，团队评估了四个方向。

**方向一：维持 Python + 优化 Streamlit**。改造成本最低，但治标不治本。Streamlit 的架构限制是框架级别的，无法通过优化绕过。Python 环境问题也无法根除。

**方向二：Python + FastAPI + 原生前端**。保留所有业务逻辑，只替换 UI 层，工作量约 1 天。但 Python 环境问题仍然存在，对开发者来说每次部署仍是负担。

**方向三：Node.js + Fastify + 原生前端（推荐方案）**。彻底替换技术栈，业务逻辑用 JavaScript 重写。工作量约 4 天后端 + 1-1.5 天前端骨架，完成后所有问题一次性解决。开发者熟悉 Node.js，未来维护和迭代效率更高。选择 Fastify 而非 Express 的具体理由见第三部分决策六。

**方向四：彻底重构为微服务**。适合产品化阶段，但对当前"单顾问现场作战工具"的定位过度设计。

最终选择**方向三**，理由是它在改造成本和长期收益之间取得最佳平衡。

## **第二部分：改造范围**

### **四、保留不变的部分**

`server.js` 作为新主入口，基于 Fastify 框架，承担原 `main_app.py` 的职责。前端通过 HTTP 调用 API、通过 WebSocket 接收实时事件。

**前端骨架（`public/` 目录）**也是 Python 版本不存在的新增产物——Streamlit 版本的 UI 由 Python 函数式声明生成，本次改造需要从零搭建独立的 HTML/CSS/JS 静态资源，包括三栏布局、WebSocket 客户端封装、演示模式 `body.classList` 切换逻辑、候选卡覆盖式弹出组件、共识链实时滚动列表组件。

为避免范围蔓延，明确以下事项**不在本次改造范围内**：

不做产品功能扩展。改造目标是技术栈替换，不是借机加新功能。所有"如果顺便做了 X 就更好了"的想法一律推迟到下一阶段。

不做 UI 重新设计。前端三栏布局、配色方案、交互流程完全保留 PRD v1.1 的产品规格，只是从 Streamlit 实现改写为静态前端实现。

不做 TypeScript 迁移。当前实施语言为 JavaScript + JSDoc 注释，TypeScript 作为独立后续任务在 Node.js 版本稳定运行 2 周后启动（见第七部分选项 D）。在此之前，关键模块使用 `// @ts-check` + JSDoc 类型注释作为过渡，编辑器层面提供类型检查和自动补全，不引入编译步骤。

不做性能优化。除了 WebSocket 替代轮询带来的天然提升外，不专门做性能优化。

## **第三部分：关键技术决策**

### **八、决策清单**

**决策一：用 `@larksuiteoapi/node-sdk` 替代 `lark-cli` 子进程调用**

当前 Python 版本通过 `subprocess.run(['lark-cli.cmd', ...])` 调用飞书命令行工具，每次调用启动一个子进程，延迟约 0.3-0.5 秒。Node.js 版本直接使用飞书官方 JS SDK，原生 HTTP 调用，延迟降至 0.1 秒以内。这是**净收益的改动**，飞书集成代码量减少约 30%，性能提升约 3 倍。

**决策二：用 `docx` npm 包替代 `python-docx`**

`docx` 是 Node.js 生态最成熟的 Word 生成库，API 风格比 `python-docx` 更现代化（基于流式构建器模式）。风险点是中文字体处理，**Day 1 下午就要单独验证**（提前于 v1.1 计划的 Day 2 上午），发现异常立即评估是否回退到 Node.js 通过 `child_process` 调用 Python `python-docx` 的混合方案，避免决策延迟挤压后续工期。

**决策三：WebSocket 替代 Streamlit 轮询**

当前飞书侧字段变更通过 30 秒轮询同步到 Streamlit。新方案中 Node.js 直接维护飞书的 WebSocket 长连接（飞书官方支持 `drive.file.bitable_record_changed_v1` 事件订阅），变更事件 1 秒内推送到前端。**Python 版本仅实现了 30 秒轮询，WebSocket 长连接从未集成进生产路径，仓库中仅保留独立测试脚本 `test-websocket-events.mjs` 作为可行性验证产物。Node.js 改造正好补齐这一能力**，这是真实交付价值，不是 Python 版本已有能力的等效替换。

**决策四：单用户本地架构不变**

当前系统设计目标是"单个顾问的现场作战工具"，全局状态用单例即可。Node.js 改造后保持这个架构。如果未来需要多顾问同时使用，再加一层会话隔离。

**决策五：演示模式不再需要"重渲染恢复"机制**

Python + Streamlit 版本的演示模式有一个核心痛点：Streamlit 重渲染会重置 CSS 类。`demo_mode.py` 通过 `st.session_state` 维护状态只能保住 Python 状态变量、保不住前端样式态，必须在每次渲染回调里重新注入 CSS 类，新增任何渲染入口都要记得补一遍——这是脆弱的绕路方案。Node.js + 静态 HTML 架构是单页应用，演示模式状态保存在 `body.classList` 上，永远不会丢失。**这是改造的隐性收益**，Streamlit 版本花大量篇幅设计的"重渲染恢复机制"在 Node.js 版本下完全不需要。

**决策六：Web 框架选用 Fastify 而非 Express**

经过 Express / Fastify / Koa / Hono / NestJS 五个方向的评估，选择 Fastify，理由如下。

第一，**async/await 原生错误冒泡**。业务模块抛出的异常自动送到全局 `setErrorHandler` 钩子，统一记录日志、调用 `fallbackHandler.js` 降级、返回结构化错误响应，不需要在每个路由手写 `try/catch`。对会议现场工具来说，"忘记一个 try/catch 导致进程崩溃"的风险在 Fastify 下从根上消除。

第二，**内置 pino 日志**。`fastify({ logger: { level: 'info', file: 'logs/app.jsonl' } })` 一行配置即完成 Day 3 计划中"结构化日志输出到 logs/app.jsonl"的需求，不需要额外安装和挂载中间件。

第三，**路由级 JSON Schema 验证**。Fastify 路由可声明 `schema: { body, response }`，请求体不符合自动返回 400，响应体结构异常框架报警。这相当于给金标准测试集的 20 个场景加了一层运行时类型保障，Day 1 写完路由就能发现结构偏差，不用等 Day 4 演练才暴露。

第四，**TypeScript 演进友好**。当前使用 JavaScript + JSDoc，后续迁移 TypeScript 时 Fastify 是一等公民（`FastifyRequest<{ Body: T }>` 泛型直接补全），迁移成本低于 Express。

Express 被排除的主要理由：async 错误处理需要 `express-async-errors` 补丁、无内置 schema 验证、无内置 pino、TypeScript 支持依赖 `@types/express` 且 `req.body` 永远是 `any`。Express 的生态广度优势在单用户本地工具场景下不兑现。

NestJS 被排除：对当前"单顾问现场作战工具"定位过度设计，依赖注入容器和 IoC 概念引入的学习成本与样板代码量会拖慢 4 天工期。

**决策七：JavaScript + JSDoc 过渡，TypeScript 作为独立后续任务**

当前实施语言为 JavaScript。在所有关键模块顶部添加 `// @ts-check`，关键函数和数据结构附加 JSDoc 类型注释，编辑器（VS Code）提供类型检查和自动补全，不引入编译步骤、不增加构建复杂度、不消耗本次改造工期。

TypeScript 完整迁移推迟到 Node.js 版本稳定运行 2 周后作为独立任务启动（见第七部分选项 D）。现在合并 TypeScript 的主要风险是：两个变量（框架迁移 + 语言迁移）同时引入时调试根因难以定位、飞书 SDK 类型声明质量未知、`strict` 模式首次迁移会集中爆发类型错误，叠加在 7 天工期上会推升至 9-12 天，置信度显著下降。

**重点实现**：

`candidateGen.js` 必须实现 `CandidateCache` 类，监听 `consensusChain` 的事件，共识链变更时异步预生成，缓存有效期 30 秒，手动修正事实时立即作废。这是 PRD 3.2 节的硬性要求，缺失会破坏 < 1.5 秒响应承诺。

`knowledgeRetriever.js` 必须实现限流：同一关键词 5 秒内不重复触发、备弹区最短刷新 30 秒、单次会议召回上限 50 次。

**前端并行轨道**：完成前端骨架与后端 API/WebSocket 的联调，覆盖候选生成、共识链推送、演示模式切换三条主路径。

**重点实现**：

`feishuClient.js` 必须实现 429 处理逻辑：读取 `x-ogw-ratelimit-reset` 响应头，等待对应秒数后重试，禁止立即重试。批量写入 500ms 间隔。`@larksuiteoapi/node-sdk` 默认不会自动处理这个，需要手动包装。

`memoGenerator.js` 的第三层 AI 润色必须串行调用各章节（不能 5 个章节并发），或在 `llmClient.js` 已有的 `p-limit(3)` 保护下调用。

**Day 2 验收**：能从 Node.js 端读写飞书多维表格、生成包含中文内容的备忘录和作战卡 Word 文件 + 跑通金标准测试集后 10 个场景 + 前端与后端联调通过。

### **十二、Day 3 HTTP 服务 + WebSocket 推送 + 持久化**

完成 `server.js` 主入口（基于 Fastify）、API 路由、WebSocket 连接管理、`fallbackHandler.js` 降级处理器、`sessionPersistence.js` 会话持久化。

**重点实现**：

`server.js` 基于 Fastify 初始化，配置内置 pino 日志直接输出到 `logs/app.jsonl`，无需额外中间件：

```javascript
// @ts-check
const fastify = require('fastify')({
  logger: { level: process.env.LOG_LEVEL || 'info', file: 'logs/app.jsonl' }
});
```

所有业务路由附加 JSON Schema 声明，确保请求/响应结构在运行时可验证。全局错误钩子统一处理业务模块异常，调用 `fallbackHandler.js` 降级后返回结构化错误响应，不依赖各路由手写 `try/catch`。

`sessionPersistence.js` 每次共识链变更时把 `chain.records` 写入 `data/session_<日期>.json`，启动时自动恢复最近一次会话。

`fallbackHandler.js` 实现真正的本地缓存——飞书 API 失败时把待写数据存到 `logs/feishu_local_cache.json`，恢复后能批量重试。LLM 超时时返回降级模板内容（不能空白）。

**Day 3 验收**：完整链路跑通 + WebSocket 推送延迟 < 1 秒 + 降级方案验证 + 服务崩溃后能从持久化文件恢复 + 日志文件可读。
